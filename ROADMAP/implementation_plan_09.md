## 1. Аудит та стабілізація API (API Audit & Swift Guidelines)

Для забезпечення відповідності стандартам мови Swift та зручності користувачів, буде проведено ретельний рефакторинг усіх публічних інтерфейсів екосистеми:

- **Відповідність API Design Guidelines:**
  - Перевірка іменування методів (використання дієслів для мутуючих методів, іменників/прикметників для немутуючих, наприклад, `sort()` vs `sorted()`).
  - Оптимізація міток аргументів (argument labels) для природного звучання коду (наприклад, `fit(features:targets:)` замість `fit(X:y:)`).
  - Чітке розділення публічного контракту та внутрішніх деталей: усе, що не призначене для кінцевого користувача, буде позначено як `internal` або `private`.
- **Сувора безпека типів та Strict Concurrency:**
  - Усі публічні типи даних, що передаються між потоками (включаючи `DataFrame`, `KMeans`, `ARIMAModel` та інше), мають бути повністю сумісні з протоколом `Sendable` під суворим наглядом Swift 6 compiler.
  - Повне очищення кодової бази від застарілих/депрекованих методів або збереження зворотної сумісності за допомогою `@available(*, deprecated, ...)` та `@_disfavoredOverload`.

## 2. Повне покриття DocC документацією

Кожен публічний символ повинен мати вичерпну та корисну документацію.

- **Формат DocC коментарів:**
  - Короткий однорядковий опис (Summary).
  - Детальний опис поведінки (Discussion).
  - Секція параметрів (`- Parameter name: Description`).
  - Опис результату повернення (`- Returns: Description`).
  - Можливі помилки (`- Throws: Description`).
- **Створення навчальних статей (Articles):**
  - `GettingStarted.md` — швидкий старт (створення DataFrame, базове тренування лінійної регресії).
  - `MigrationGuide.md` — гайд з переходу з Python/Scikit-Learn на SwiftAnalytics.
  - `ArchitectureOverview.md` — пояснення Hardware Routing та UMA optimisation.
- **GitHub Actions Integration:**
  - Налаштування автоматичного збирання та деплою DocC-архіву на GitHub Pages при кожному пуші до гілки `main`.

## 3. Swift Package Index конфігурація (.spi.yml)

Для інтеграції із провідним каталогом Swift-пакетів [Swift Package Index](https://swiftpackageindex.com) буде створено файл `.spi.yml` у корені проєкту:

- **Параметри конфігурації:**
  - Підтримка збірки документації для всіх модулів.
  - Визначення цільових платформ для авто-перевірки сумісності (macOS, iOS, visionOS, Linux).
  - Додавання метаданих для відображення badges.

```yaml
version: 1
builder:
  configs:
    - platform: macos
      swift_version: "6.0"
    - platform: ios
      swift_version: "6.0"
```

## 4. Семантичне версіонування та CHANGELOG

Встановлення правил SemVer 2.0 для подальшого життєвого циклу проєкту:

- **Формат версій:** `MAJOR.MINOR.PATCH`.
  - `MAJOR` — несумісні зміни API.
  - `MINOR` — нова функціональність зі збереженням зворотної сумісності.
  - `PATCH` — виправлення помилок (bug fixes) без додавання функцій.
- **Документування змін:**
  - Створення `CHANGELOG.md` у корені проєкту за форматом "Keep a Changelog".
  - Історичний запис усіх значних досягнень з версії v0.0 до v0.9 (включаючи реліз Concurrency, SwiftForecast та Hardware Routing).

## 5. Розширення модуля SwiftNLP

Модуль `SwiftNLP` буде значно розширено для підготовки бази майбутнього інтелектуального модуля `SwiftLLM` (v1.0).

### 5.1 Протокол Tokenizer та BPETokenizer

Буде визначено уніфікований інтерфейс токенізації тексти:

```swift
public protocol Tokenizer: Sendable {
    /// Розбиває текст на масив токенів (рядків).
    func tokenize(text: String) -> [String]
    
    /// Кодує текст у послідовність цілих чисел (ID токенів).
    func encode(text: String) -> [Int]
    
    /// Декодує ID токенів назад у вихідний рядок.
    func decode(tokens: [Int]) -> String
}
```

Буде реалізовано `BPETokenizer` (Byte Pair Encoding) на чистому Swift:
- **Характеристики:**
  - Підтримка завантаження файлів словника `vocab.json` (Token -> ID) та правил злиття `merges.txt`.
  - Збереження стану токенізатора з можливістю гарячого перемикання словників.
  - Ефективний пошук злиття найчастіших байтових пар.
  - Стійкість до OOV (Out-of-Vocabulary) слів шляхом розбиття до байтового рівня.

### 5.2 WordEmbeddings (Word2Vec/GloVe lookup)

Реалізація класу `WordEmbeddings` для роботи з дистрибутивними векторами:
- **Функціонал:**
  - Завантаження претренованих статичних ембеддінгів з текстових файлів (формат: `word float1 float2 ... floatN`).
  - Метод `vector(for word: String) -> [Double]?`.
  - Розрахунок косинусної близькості (`cosineSimilarity`) між двома словами.
  - Пошук найближчих слів до заданого вектора (Top-K query).
  - Безпечний конкурентний доступ до матриці ембеддінгів (акторська ізоляція).

## 6. План верифікації та тестування

### 6.1 Автоматичні тести
Будуть додані наступні тестові сценарії в `Tests/SwiftNLPTests`:
- **BPETokenizerTests:**
  - Тестування токенізації та кодування/декодування простих речень.
  - Перевірка кругового циклу (Round-trip) `decode(encode(text)) == text`.
  - Валідація обробки невідомих символів та спецсимволів (BOS/EOS/PAD).
- **WordEmbeddingsTests:**
  - Тестування завантаження файлу векторів слів.
  - Тест розрахунку схожості слів (наприклад, `similarity("king", "queen") > similarity("king", "apple")`).
  - Тест Top-K пошуку найближчих векторів.

### 6.2 Документаційний аудит
- Локальне збирання DocC документації за допомогою команди:
  `swift package --allow-writing-to-directory-path ./docs generate-documentation --target SwiftAnalytics`
- Перевірка відсутності незадокументованих публічних символів (`-warnings-as-errors` під час збірки документації).
