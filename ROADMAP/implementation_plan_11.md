## 1. Асинхронний потоковий CSV-парсер (`SwiftDataFrame`)

### Проблема:

Наразі CSV-парсер у `SwiftDataFrame` зчитує весь файл у пам'ять цілком і обробляє його одним великим блоком, що створює пляшку ($O(N)$ CPU-навантаження на один потік). Це призводить до 10-кратного відставання від Pandas на файлах розміром від 100k рядків (169 ms у Swift проти 18 ms у Pandas).

### Рішення:

Впровадити **Chunk-based Streaming Parser** на базі Swift `AsyncSequence`.

#### Архітектурні деталі:

1. **Буферизоване зчитування (`BufferedReader`)**: Зчитування сирого файлу з диска фіксованими чанками (наприклад, по 64 KB чи 256 KB) з використанням низькорівневого POSIX API (`read` / `pread`) або `FileHandle`.
2. **Паралельне розпізнавання типів (`Type Inference`)**:
   - Розбиття сирого буфера на рядки та стовпчики.
   - Використання Swift `TaskGroup` для асинхронного парсингу окремих чанків у паралельних потоках.
   - Потоковий збір результату безпосередньо в блоки `Apache Arrow` (zero-copy allocation).
3. **Асинхронний інтерфейс**:
   ```swift
   public extension DataFrame {
       static func readCSVStream(
           contentsOf url: URL,
           chunkSize: Int = 10_000
       ) -> AsyncThrowingStream<DataFrame, Error> {
           // Повертає DataFrame частинами (чанками) по мірі зчитування
       }
   }
   ```

### Очікувані критерії успіху:

- Час зчитування 100k рядків має зменшитись до **< 40 ms** (прискорення в 4 рази).
- Ефективне споживання пам'яті (пікове споживання не повинно перевищувати розмір файлу більш ніж у 1.5 рази завдяки потоковій обробці).

## 2. Парсер SafeTensors та GGUF (`SwiftLLM`)

### Проблема:

Зараз `SwiftLLM` вимагає попередньої конвертації ваг моделей у проміжний формат за допомогою Python-скриптів. Це ускладнює інтеграцію моделей Hugging Face у Swift-додатки.

### Рішення:

Створити нативний парсер форматів **SafeTensors** та **GGUF** на Swift.

#### Архітектурні деталі:

1. **SafeTensors Parser**:
   - Формат SafeTensors складається з 8-байтового заголовка (довжина JSON-метаданих) та самого JSON-заголовка, за яким слідують сирі байти тензорів.
   - Нативний парсер зчитуватиме JSON-заголовок, витягуватиме метадані про типи даних (F32, F16, BF16) та зміщення (offsets).
   - Відображення байти-в-масив через `MLXArray` без дублювання пам'яті (zero-copy memory mapping за допомогою `Data(bytesNoCopy:)`).
2. **GGUF Parser**:
   - Зчитування заголовка GGUF (магічне число, версія, кількість тензорів, кількість метаданих).
   - Парсинг словника метаданих (key-value pairs) та таблиці тензорів.
   - Підтримка деквантизації ваг на льоту або передача квантованих даних напряму в GPU Metal-ядра.

### Перевірки та тести:

- Тест на правильність завантаження ваг невеликої моделі (наприклад, TinyLlama або MobileLLM).
- Перевірка відповідності вихідних токенів (logits) після прямого проходу з вагами, завантаженими через SafeTensors, порівняно з еталонними вагами.

## 3. Схема CKKS у `SwiftPrivacy`

### Проблема:

Схема Ring-LWE, що реалізована зараз, працює лише з цілими числами. Для більшості аналітичних завдань (наприклад, обчислення середнього, дисперсії чи прогнозів моделей регресії на зашифрованих даних) необхідна підтримка дійсних чисел (дробів).

### Рішення:

Впровадити схему **CKKS (Cheon-Kim-Kim-Song)** для наближеного гомоморфного шифрування дійсних чисел.

#### Архітектурні деталі:

1. **Кодування з фіксованою комою (Fixed-Point Encoder)**:
   - Масштабування дійсного числа $x$ за допомогою коефіцієнта $\Delta = 2^p$ перед перетворенням у поліном.
   - Проєкція на круговий поліном (cyclotomic polynomial).
2. **Rescaling (Масштабування)**:
   - Після кожного гомоморфного множення шифротекстів масштаб збільшується до $\Delta^2$.
   - Потрібно реалізувати процедуру `rescale`, яка ділить шифротекст на $\Delta$ та оновлює рівень шифрування (modulus reduction).
3. **Гомоморфні операції**:
   - `add(cipherA, cipherB)` -> Ciphertext
   - `multiply(cipherA, cipherB)` -> Ciphertext

### Перевірки та тести:

- Тест на точність обчислень: порівняння результату розшифрування $(a \cdot b + c)$ на схемі CKKS із класичним Float-обчисленням. Допустима похибка (noise) має бути $< 10^{-5}$.

## 4. SARIMA та GARCH у `SwiftForecast`

### Проблема:

Клас `ARIMAModel` наразі підтримує лише прості лінійні тренди та стаціонарні процеси без сезонних коливань або зміни волатильності в часі.

### Рішення:

Додати класи **`SARIMAModel`** та **`GARCHModel`**.

#### Архітектурні деталі:

1. **SARIMA (Seasonal ARIMA)**:
   - Підтримка параметрів $(p, d, q) \times (P, D, Q)_s$.
   - Сезонне диференціювання рядів: $y_t - y_{t-s}$ за допомогою `vDSP_vsub`.
   - Оцінка параметрів методом Conditional Least Squares з розширенням регресійної матриці OLS для сезонних лагів.
2. **GARCH(1,1) (Generalized Autoregressive Conditional Heteroskedasticity)**:
   - Моделювання умовної дисперсії: $\sigma_t^2 = \omega + \alpha \epsilon_{t-1}^2 + \beta \sigma_{t-1}^2$.
   - Визначення залишків моделі (residuals) та чисельна максимізація функції логарифмічної правдоподібності (Maximum Likelihood Estimation) через градієнтний спуск або BFGS.

### Перевірки та тести:

- Тест на прогнозування штучного сезонного ряду (період = 12) з оцінкою точності (MAPE < 5%).
- Перевірка обчислення умовної волатильності на фінансових рядах (акції/індекси).
